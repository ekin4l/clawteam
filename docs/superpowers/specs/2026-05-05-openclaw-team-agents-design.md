# OpenClaw Team Agents Configuration Design

**Date:** 2026-05-05
**Status:** Draft
**Version:** 1.0

---

## 1. Overview

Configure 5 independent OpenClaw agent instances, each representing a specialized role in a software development team. The agents collaborate with human engineers via Feishu, integrate with CodeArts for task management, and operate under a layered configuration system that separates concerns: roles, work items, tools, team knowledge, and compliance rules.

### Core Design Principles

- **Decoupling**: Roles, work items, tools, team info, and compliance are independently configured and managed
- **Role-key binding**: Memory and persistent data bind to `role_key` (e.g., `assistant`), NOT to display names (e.g., "Nora")
- **Template-driven**: YAML source files + Jinja2 templates render into OpenClaw native workspace files
- **Instance-aware**: Pre-flight check detects actual OpenClaw paths (supports cloud vendor variants)
- **Incremental updates**: Re-render config files without touching agent-generated memory

## 3. Configuration File Specifications

### 3.1 Agent Definition (`agents/*.yaml`)

Each agent file contains the complete role definition. Filename format: `{role_key}_{display_name}.yaml`.

```yaml
# agents/assistant_nora.yaml
role_key: assistant                    # Permanent identifier, memory/routes bind to this
display_name: "Nora"                   # Configurable display name
role_title: "研发助理"                  # Role title shown in communications
creature: "AI assistant"               # Identity type
emoji: "🌸"
vibe: "warm, organized, proactive"     # Communication style
avatar: "avatars/nora.svg"             # Avatar file path
model: "zai/glm-4.7"                   # LLM model for this agent
communicates_externally: true          # Whether this agent talks directly to humans

personality: |
  你是 Nora，一位温暖而高效的研发助理。你善于倾听、善于整理信息，
  是团队沟通的枢纽。你主动但不强势，简洁但不冷漠。
  你像一位经验丰富的项目协调人。

assigned_work_items:                    # Work items this agent handles
  - daily_report
  - meeting_minutes
  - requirement_analysis
  - task_registration
  - progress_tracking
  - doc_management
  - group_facilitation

heartbeat_tasks:                        # Periodic tasks for HEARTBEAT.md
  - "工作日 18:00 收集团队日报"
  - "每日 09:00 检查任务进度并预警"

capabilities:                           # What this agent can do
  can_create_feishu_groups: true
  can_invite_to_groups: true
  can_send_feishu_messages: true
  can_create_codearts_issues: true
  can_manage_codearts_repo: true
```

### 3.2 Work Item Definition (`work_items/*.yaml`)

Each work item is a standalone file for easy management and extension.

```yaml
# work_items/daily_report.yaml
name: daily_report
display_name: 日报收集整理
description: 自动收集团队成员工作日报并整理成结构化报告
category: communication        # communication | development | quality | ops | testing
approval_level: low            # low (auto) | medium (notify) | high (approve)

tools:                         # Required tools
  - feishu/messaging
  - feishu/group_management

triggers:                      # How this work item is triggered
  scheduled: "0 18 * * 1-5"   # Weekdays 18:00
  manual: "@Nora 日报"          # Manual trigger phrase

inputs:                        # Expected inputs
  - 团队成员飞书消息
outputs:                       # Expected outputs
  - 结构化日报（飞书群消息）
```

### 3.3 Tool Definition (`tools/**/*.yaml`)

Tools are organized by external system.

```yaml
# tools/codearts/issue_crud.yaml
name: codearts_issue_crud
display_name: CodeArts Issue 管理
category: codearts
description: 创建、更新、查询、关闭 CodeArts 工作项
approval_level: medium

operations:
  - create_issue
  - update_issue
  - query_issues
  - add_comment
  - close_issue
  - assign_issue

api_config:
  connector: codearts
  base_url_ref: "${CODEARTS_BASE_URL}"
  auth_type: api_token
```

  avoid_topics:
    - "不讨论公司财务数据"

### 3.6 Assignments (`assignments.yaml`)

```yaml
# assignments.yaml
defaults:
  assistant:
    - daily_report
    - meeting_minutes
    - requirement_analysis
    - task_registration
    - progress_tracking
    - doc_management
    - group_facilitation

  architect:
    - requirement_analysis       # Shared: both assistant and architect can do
    - architecture_design

  quality:
    - code_review
    - review_report

  tester:
    - test_case_generation
    - test_automation
    - test_execution
    - defect_reporting

  ops:
    - deploy_review
    - risk_assessment
    - ops_documentation
    - deploy_execution

collaboration:
  - trigger: code_review         # Ken finds issues
    action: group_facilitation   # Nora creates group to discuss
    agents: [quality, assistant]

  - trigger: architecture_design # Mark completes design
    action: test_case_generation # Alice starts generating tests
    agents: [architect, tester]

  - trigger: deploy_execution    # Emily about to deploy
    action: risk_assessment      # Requires approval first
    agents: [ops]
    requires_approval: true
```

### 3.7 Instance Config (`instance.yaml`)

Auto-generated by `preflight-check.sh`. Do not manually edit.

```yaml
# instance.yaml
detected_at: "2026-05-05"
openclaw:
  version: "2026.5.4"
  variant: "standard"                    # standard | huawei | aliyun | ...
  state_dir: "/Users/norman/.openclaw"
  config_path: "/Users/norman/.openclaw/openclaw.json"
  workspace_root: "/Users/norman/.openclaw/workspace"
  agents_dir: "/Users/norman/.openclaw/agents"

capabilities:
  multi_agent: true
  agents_add: true
  agents_bind: true
  agents_set_identity: true

agents:
  existing:
    - role_key: main
      workspace: "/Users/norman/.openclaw/workspace"
      model: "zai/glm-4.7"
```

## 4. Template Rendering

### 4.1 Rendering Flow

```
agents/assistant_nora.yaml  ─┐
work_items/*.yaml           ─┤
tools/**/*.yaml             ─┤  render-workspace.py   Target OpenClaw workspace
team/*.yaml                 ─┤  ──────────────────>  (paths from instance.yaml)
compliance.yaml             ─┤
assignments.yaml            ─┤
templates/SOUL.md.tmpl      ─┘
```

### 4.2 Output Files per Agent

Each agent workspace receives these rendered files:

| File | Source | Update Policy |
|------|--------|---------------|
| `SOUL.md` | personality + compliance rules | Overwrite on update |
| `AGENTS.md` | operations + assigned work items + collaboration rules + compliance | Overwrite on update |
| `IDENTITY.md` | display_name, emoji, avatar, creature | Overwrite on update |
| `HEARTBEAT.md` | heartbeat_tasks from agent config | Overwrite on update |
| `TOOLS.md` | Base template + tools assigned to this agent | Merge: preserve agent notes |
| `TEAM.md` | All team/*.yaml rendered | Overwrite on update |
| `MEMORY.md` | — | **Never touch** (agent-owned) |
| `USER.md` | — | **Never touch** (agent-owned) |
| `memory/` | — | **Never touch** (agent-owned) |

### 4.3 Update Policies

- **Overwrite**: Template content replaces existing file. Used for config files we own.
- **Merge (TOOLS.md)**: Preserve agent-added notes, append/update template sections.
- **Never touch**: MEMORY.md, USER.md, memory/ — agent-owned runtime data.

## 5. Script Specifications

### 5.1 `preflight-check.sh`

**Purpose**: Detect OpenClaw instance configuration and generate `instance.yaml`.

**Steps**:
1. Run `openclaw --version` to detect version
2. Run `openclaw config file` to find config path
3. Probe directory structure: state_dir, workspace_root, agents_dir
4. Check `openclaw agents list` for existing agents
5. Test capabilities: `agents add`, `agents bind`, `agents set-identity`
6. Write `instance.yaml`

**Usage**:
```bash
scripts/preflight-check.sh                    # Auto-detect
scripts/preflight-check.sh --state-dir /path  # Override state dir
```

### 5.2 `setup-agents.sh`

**Purpose**: Create and configure all agents from YAML definitions.

**Steps**:
1. Read `instance.yaml` for paths and capabilities
2. For each agent in `agents/` directory:
   a. If `--target <existing_agent>` specified: render workspace files into existing agent's workspace
   b. Otherwise: run `openclaw agents add <role_key> --workspace <path> --model <model>`
   c. Render workspace files via `render-workspace.py`
   d. Run `openclaw agents set-identity` if supported
3. Configure routing bindings if specified

**Usage**:
```bash
scripts/setup-agents.sh --all                          # Create all 5 agents
scripts/setup-agents.sh assistant                      # Create only assistant
scripts/setup-agents.sh assistant --target main        # Apply assistant config to existing main agent
```

### 5.3 `update-agent.sh`

**Purpose**: Incrementally update an agent's config without touching memory.

**Steps**:
1. Read agent definition from `agents/assistant_nora.yaml`
2. Re-render workspace files (SOUL.md, AGENTS.md, etc.)
3. Apply update policy per file type (overwrite/merge/never touch)
4. Skip MEMORY.md, USER.md, memory/

**Usage**:
```bash
scripts/update-agent.sh assistant                      # Update by role_key
scripts/update-agent.sh --all                          # Update all agents
scripts/update-agent.sh assistant --target main        # Update specific target
```

### 5.4 `add-agent.sh`

**Purpose**: Interactively add a new role (e.g., full-stack engineer).

**Steps**:
1. Prompt for role_key, display_name, role_title, etc.
2. Copy `_template.yaml` and fill in values
3. Optionally assign work items
4. Create agent via `openclaw agents add`
5. Render workspace files

**Usage**:
```bash
scripts/add-agent.sh                    # Interactive mode
scripts/add-agent.sh --name "dev_fullstack_liam.yaml" --non-interactive
```

### 5.5 `remove-agent.sh`

**Purpose**: Remove an agent and optionally backup memory.

**Steps**:
1. Export memory first (prompt user)
2. Run `openclaw agents delete <role_key>`
3. Remove from `instance.yaml`

**Usage**:
```bash
scripts/remove-agent.sh assistant              # Remove with memory backup prompt
scripts/remove-agent.sh assistant --keep-config  # Keep YAML definition, only remove OpenClaw instance
```

### 5.6 `render-workspace.py`

**Purpose**: Jinja2 template engine that renders workspace files from YAML sources.

**Inputs**:
- Agent YAML (e.g., `agents/assistant_nora.yaml`)
- Assigned work items (resolved from `assignments.yaml` -> `work_items/*.yaml`)
- Required tools (resolved from work items -> `tools/**/*.yaml`)
- Team info (`team/*.yaml`)
- Compliance rules (`compliance.yaml`)
- Instance paths (`instance.yaml`)

**Outputs**: Rendered .md files written to the agent's workspace directory.

**Usage**:
```bash
scripts/render-workspace.py --agent assistant --output /path/to/workspace
scripts/render-workspace.py --all
```

### 5.7 `export-memory.sh`

**Purpose**: Export an agent's memory for backup or migration.

**Steps**:
1. Read workspace path from `instance.yaml`
2. Copy MEMORY.md, USER.md, memory/ to `memories/<role_key>/`
3. Optionally create tar.gz archive

**Usage**:
```bash
scripts/export-memory.sh assistant                           # Export to memories/assistant/
scripts/export-memory.sh assistant --output ./backup.tar.gz  # Export as archive
scripts/export-memory.sh --all --output ./all-memories.tar.gz
```

### 5.8 `import-memory.sh`

**Purpose**: Import memory into an agent's workspace.

**Steps**:
1. Validate target agent exists
2. Copy MEMORY.md, USER.md, memory/ from source to agent workspace
3. Skip if target already has memory (prompt user for merge/overwrite)

**Usage**:
```bash
scripts/import-memory.sh assistant --input ./backup.tar.gz
scripts/import-memory.sh assistant --source memories/assistant/
```

## 6. Workspace File Contents

### 6.1 SOUL.md Template

The rendered SOUL.md for each agent contains:
- **Core identity**: personality, role, communication style
- **Boundaries**: from compliance.yaml (avoid topics, content policy)
- **Vibe**: agent-specific communication tone
- **Continuity**: how to maintain state across sessions

### 6.2 AGENTS.md Template

The rendered AGENTS.md for each agent contains:
- **Session startup routine**: which files to read
- **Assigned work items**: detailed descriptions, triggers, tools
- **Collaboration rules**: when to involve other agents
- **Tool usage guidelines**: from tools/*.yaml for assigned tools
- **Safety rules**: from compliance.yaml
- **Memory management**: how to maintain daily logs and MEMORY.md
- **Group chat behavior**: platform-specific rules

### 6.3 IDENTITY.md Template

```markdown
# IDENTITY.md - Who Am I?

- **Name:** {{ display_name }}
- **Creature:** {{ creature }}
- **Vibe:** {{ vibe }}
- **Emoji:** {{ emoji }}
- **Avatar:** {{ avatar }}
```

### 6.4 HEARTBEAT.md Template

```markdown
# HEARTBEAT.md

{% for task in heartbeat_tasks %}
- [ ] {{ task }}
{% endfor %}
```

### 6.5 TOOLS.md Template

Base template + agent-specific tools rendered from `tools/`. Agent's own notes are preserved during updates.

### 6.6 TEAM.md Template

All team members rendered into a reference document:

```markdown
# TEAM.md - Team Members

{% for member in team %}
## {{ member.name }}
- **Role:** {{ member.role }}
- **Responsibilities:** {{ member.responsibilities | join(', ') }}
- **Goals:** {{ member.goals | join(', ') }}
- **Modules:** {{ member.modules | join(', ') }}
{% endfor %}
```

## 7. Approval Levels

| Level | Behavior | Examples |
|-------|----------|----------|
| **low** | Auto-execute, no confirmation needed | Daily reports, meeting minutes, docs |
| **medium** | Auto-execute, notify relevant people | CodeArts issue creation, code review reports |
| **high** | Must get human approval before executing | Production deployment, deleting resources |

Work items define their approval level. Agents respect these levels in their AGENTS.md instructions.

## 8. Trigger Types

Agents support 4 trigger mechanisms:

| Trigger | Mechanism | Example |
|---------|-----------|---------|
| **Feishu @mention** | `openclaw agents bind` routes @mentions to agent | `@Nora 帮我创建一个任务` |
| **CodeArts event** | Webhook triggers agent action | New MR triggers code review |
| **Scheduled** | HEARTBEAT.md / cron tasks | Daily 18:00 report collection |
| **Git event** | Repository webhook triggers agent | Push triggers test execution |

## 9. Future Extensibility

The template-based design supports easy extension:

### Adding a new role (e.g., Full-Stack Engineer)

1. Create `agents/dev_fullstack_liam.yaml` from `_template.yaml`
2. Create any new work items in `work_items/`
3. Update `assignments.yaml` to assign work items
4. Run `scripts/setup-agents.sh dev_fullstack`

### Adding a new tool (e.g., Jira)

1. Create `tools/jira/` directory with tool definitions
2. Reference in relevant work items
3. Run `scripts/update-agent.sh --all`

### Adding a new team member

1. Create `team/new_member.yaml` from `_template.yaml`
2. Run `scripts/update-agent.sh --all` to update all agents' TEAM.md

## 10. Execution Flow Summary

```
Step 1: scripts/preflight-check.sh        → instance.yaml
Step 2: scripts/setup-agents.sh --all     → 5 OpenClaw agents created + workspace files rendered
Step 3: Daily operations via Feishu/Git/CodeArts triggers
Step 4: As needed: update-agent.sh, export-memory.sh, import-memory.sh
```
